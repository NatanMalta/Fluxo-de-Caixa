# PIN-based auth for LAN API

A aplicação é de usuário único e foi originalmente concebida sem nenhum tipo de auth — o glossário dizia textualmente "Não há tela de login". Como o backend escuta em `0.0.0.0:5000` na LAN da loja, qualquer dispositivo na mesma rede Wi-Fi consegue hoje chamar a API direto. Para defesa mínima nesse cenário, decidimos adicionar um esquema de autenticação baseado em PIN único: o dono configura um PIN numérico curto em `appsettings.json`, o app Flutter usa esse PIN como tela de bloqueio na abertura e como credencial para obter um JWT de 30 dias em um novo endpoint `POST /api/auth/login`, e todas as chamadas seguintes à API exigem `Authorization: Bearer <jwt>`.

## Considered options

- **ASP.NET Core Identity** (`SignInManager`, `UserStore`, `AspNetUsers`, `PasswordHasher`, policies, lockout, 2FA, external providers): rejeitado. Foi desenhado para cenários com milhares de usuários, papéis e senhas com regras de complexidade. Para 1 usuário / 1 PIN / 1 LAN, é canhão em formiga. Cada `dotnet new` que adiciona uma migração do Identity arrasta tabelas (`AspNetRoleClaims`, `AspNetUserTokens`, `AspNetUserLogins`...) que este projeto nunca vai consultar.
- **PIN em todo request** (`X-Pin: 1234` em cada chamada à API): rejeitado. O PIN viaja pela LAN em **toda** chamada e aparece em qualquer log de proxy, HAR do navegador, trace do APM, etc. Mesmo a gente nunca ligando APM, qualquer `dotnet user-secrets` mal feito ou print de debug vaza.
- **Chave de API estática no header** (igual ao item anterior, só que com string longa aleatória em vez de PIN): rejeitado. Mesma classe de problema do item anterior (segredo em todo request) e pior UX (o dono não consegue digitar uma string longa no celular).
- **Hash do PIN em `appsettings.json` + ferramenta CLI para gerar (`dotnet run --setup-pin`)**: considerado e descartado pelo usuário, que preferiu editar o `appsettings.json` diretamente com o PIN em texto puro. Trade-off aceito: o PIN em claro fica no arquivo de configuração, mas isso é mitigado pelo fato de o app rodar em LAN local, sem sincronização com a nuvem, e o arquivo `appsettings.json` ser trivialmente excluível de backup na rotação do PIN.
- **Persistir o PIN em DB (SQLite) com migration**: rejeitado. Mistura config (PIN = configuração) com estado (lançamentos, contas, etc.) sem motivo. O backend já lê o PIN no startup; colocar no banco só adiciona uma tabela e uma rota de admin para trocá-lo, que ninguém vai usar.

## Consequences

- **Novo endpoint**: `POST /api/auth/login` com body `{ "pin": "..." }` → `{ "token": "...", "expiresAt": "..." }`. Sem refresh token; o app reabre o PIN quando o JWT expira.
- **Novo middleware**: `UseAuthentication()` + `UseAuthorization()` no `Program.cs`, com handler JWT bearer. Segredo de assinatura e dias de validade lidos de `appsettings.json` (`"Jwt": { "Secret": "...", "ExpiryDays": 30 }`).
- **Migração dos controllers existentes**: adicionar `[Authorize]` em todos (ou definir uma policy global `RequireAuthenticatedUser()` e aplicá-la em `Program.cs` com `options.Filters.Add(new AuthorizeAttribute())`). Mudança mecânica, feita uma vez.
- **Novas dependências NuGet** no backend: `Microsoft.AspNetCore.Authentication.JwtBearer` (já vem no template ASP.NET 10 webapi, mas verificar) e `BCrypt.Net-Next` (para hashear o PIN em memória no startup).
- **Novas seções em `appsettings.json`**: `Auth.Pin` (texto puro) e `Jwt.Secret` + `Jwt.ExpiryDays`. O `Auth.Pin` precisa ser definido para o backend subir; sem ele, o app falha no startup com mensagem clara.
- **Rate limiter no `/api/auth/login`**: `Microsoft.AspNetCore.RateLimiting` nativo do .NET 7+, configurado como "fixed window de 5 requisições/minuto por IP". Sem estado persistente, sem DB. O atacante brute-forçando da LAN esbarra no 429 em segundos; o dono que errar 3-4 vezes digitando o PIN no celular espera no máximo 1 minuto.
- **Novas peças no Flutter**: widget de "PIN lock screen" exibido no `MaterialApp` raiz enquanto `ApiClient.token == null`; `AuthService.login(pin)` chamando `/api/auth/login`; interceptor no `ApiClient` que, em 401, limpa o token e re-exibe a tela de PIN.
- **Limitação aceita**: trocar o PIN no `appsettings.json` e reiniciar o backend **não invalida imediatamente os JWTs já emitidos** que ainda estão dentro da janela de 30 dias. Eles continuam válidos até a expiração natural. Para v1, isso é aceitável (dono único, troca de PIN é rara, max. 30 dias de "vazamento" do PIN antigo). Se incomodar, adicionar claim `pinVersion` no payload do JWT e comparar com `appsettings.json` no middleware — mudança pequena, mas optamos por adiar.
- **Limitação aceita**: o app Flutter web armazena o JWT apenas em variável em memória no `ApiClient`. Fechar a aba do navegador descarta o token, e o próximo acesso pede o PIN de novo. É a única opção de armazenamento que não mente sobre segurança no web (a implementação "web" do `flutter_secure_storage` cai para `localStorage` criptografado só com a chave ofuscada no bundle JS, visível no DevTools).
- **CORS continua aberto** para qualquer origem. Com o JWT no header, a origem do request não é mais um sinal de confiança útil; restringir CORS só esconderia o vetor, não o mitigaria. Manter aberto simplifica testes via Postman/curl.
